Implementação do projeto DRONE_INSPETOR

Guia completo de instalação em Ubuntu 24.04 — do zero até a simulação do drone de inspeção rodando.

Ubuntu 24.04 PX4 v1.16 (94bbd2d) ROS 2 Jazzy Gazebo Harmonic Micro-XRCE-DDS

00 Visão geral & pré-requisitos

O projeto integra quatro grandes componentes que serão instalados nesta ordem: o firmware PX4 (controladora de voo em SITL), o ROS 2 Jazzy (middleware de robótica), o Micro-XRCE-DDS Agent (ponte entre PX4 e ROS 2) e os pacotes do projeto (visão computacional, missão e estação de controle).

Pré-requisitos Ubuntu 24.04 (nativo ou WSL2), placa de vídeo com drivers atualizados (o Gazebo usa aceleração gráfica), acesso à internet e cerca de 40 GB livres de disco. Recomenda-se GPU NVIDIA para a renderização e para acelerar a visão computacional (veja o passo 08).
Dica Rode cada bloco de comandos de cima para baixo. Vários passos pedem reinício ou novo terminal — respeite esses avisos, pois eles carregam variáveis de ambiente necessárias.

01 Atualizar o sistema

Antes de tudo, atualize a lista de pacotes e o sistema para evitar conflitos de dependências durante a instalação.

bash
# Atualiza índices de pacotes e aplica todas as atualizações disponíveis
sudo apt update && sudo apt upgrade -y

02 Compilação e instalação do firmware PX4

Segue o procedimento oficial (docs.px4.io), porém com a versão fixada do PX4 para garantir reprodutibilidade — é a mesma da máquina de desenvolvimento.

Importante Não use a última main do PX4. O projeto depende do commit 94bbd2d6… (PX4 v1.16). Versões mais novas quebram o build e o airframe 4030.

Clonar e fixar a versão

bash
# Git é necessário para clonar o repositório
sudo apt install git -y

# Clona o PX4 já com os submódulos (--recursive)
cd
git clone https://github.com/PX4/PX4-Autopilot.git --recursive

# Fixa o commit exato usado no projeto (fica em "detached HEAD" — é esperado)
cd PX4-Autopilot
git checkout 94bbd2d69ade182bed087e8254515bb5f559285b

# Realinha TODOS os submódulos para o estado desse commit
git submodule update --init --recursive

Instalar dependências de build

bash
# Script oficial do PX4 instala toolchain, dependências e configura o ambiente
# (executado de DENTRO da pasta PX4-Autopilot)
bash ./Tools/setup/ubuntu.sh
Correção No rascunho original o caminho era ./PX4-Autopilot/Tools/setup/ubuntu.sh, mas como você já está dentro de PX4-Autopilot, o caminho correto é ./Tools/setup/ubuntu.sh.
bash
# Reinicie o sistema para aplicar grupos de usuário e variáveis do ambiente
sudo systemctl reboot

Compilar o firmware

bash
# Após reiniciar, compila o firmware PX4 para simulação (SITL)
cd ~/PX4-Autopilot
make px4_sitl
bash
# Teste rápido: inicia uma simulação Gazebo com um drone x500 simples
# (feche com Ctrl+C após confirmar que abre sem erros)
make px4_sitl gz_x500
Dica Este primeiro make px4_sitl também gera os plugins do Gazebo que serão copiados no passo 06. Não pule a compilação.

03 Instalação do ROS 2 Jazzy

Procedimento oficial para o Jazzy (compatível com Ubuntu 24.04): docs.ros.org/jazzy.

Configurar o locale (UTF-8)

bash
# Verifica se o locale já suporta UTF-8
locale

# Instala e gera um locale UTF-8
sudo apt update && sudo apt install locales
sudo locale-gen en_US en_US.UTF-8
sudo update-locale LC_ALL=en_US.UTF-8 LANG=en_US.UTF-8
export LANG=en_US.UTF-8

# Confirma a configuração
locale

Adicionar o repositório do ROS 2

bash
# Habilita o repositório "universe" do Ubuntu
sudo apt install software-properties-common -y
sudo add-apt-repository universe

# Instala o pacote que adiciona a fonte APT oficial do ROS 2
sudo apt update && sudo apt install curl -y
export ROS_APT_SOURCE_VERSION=$(curl -s https://api.github.com/repos/ros-infrastructure/ros-apt-source/releases/latest | grep -F "tag_name" | awk -F'"' '{print $4}')
curl -L -o /tmp/ros2-apt-source.deb "https://github.com/ros-infrastructure/ros-apt-source/releases/download/${ROS_APT_SOURCE_VERSION}/ros2-apt-source_${ROS_APT_SOURCE_VERSION}.$(. /etc/os-release && echo ${UBUNTU_CODENAME:-${VERSION_CODENAME}})_all.deb"
sudo dpkg -i /tmp/ros2-apt-source.deb

Instalar o ROS 2 e o bridge do Gazebo

bash
# Ferramentas de desenvolvimento (colcon, rosdep, etc.)
sudo apt update && sudo apt install ros-dev-tools -y

# ROS 2 Jazzy completo (inclui RViz, demos, tf, etc.)
sudo apt install ros-jazzy-desktop -y

# Ponte ROS 2 <-> Gazebo (ros_gz_bridge, ros_gz_image, ...)
sudo apt install ros-jazzy-ros-gz -y
Complemento O ros-jazzy-ros-gz traz o ros_gz_image, que faltava no ambiente inicial e causava o erro package 'ros_gz_image' not found no launch.

Carregar o ROS 2 automaticamente

bash
# Carrega o ROS 2 no terminal atual
source /opt/ros/jazzy/setup.bash

# Torna o carregamento permanente em todo novo terminal
echo 'source /opt/ros/jazzy/setup.bash' >> ~/.bashrc
Nota No rascunho original isso era feito editando o .bashrc pelo nano. O comando echo … >> ~/.bashrc faz o mesmo de forma automática (a seta dupla >> acrescenta ao final do arquivo, sem apagar o conteúdo).

Testar a comunicação (talker / listener)

Abra dois terminais novos e rode um comando em cada. O listener deve receber as mensagens do talker.

bash — terminal A
ros2 run demo_nodes_cpp talker
bash — terminal B
ros2 run demo_nodes_py listener

04 Instalar o Middleware (Micro-XRCE-DDS Agent)

É a ponte de comunicação que leva os tópicos internos do PX4 (uORB) para o ROS 2 via DDS.

bash
# Clona, compila e instala o Agent
cd
git clone https://github.com/eProsima/Micro-XRCE-DDS-Agent.git
cd Micro-XRCE-DDS-Agent
mkdir build
cd build
cmake ..
make
sudo make install

# Atualiza o cache de bibliotecas compartilhadas
sudo ldconfig /usr/local/lib/
bash
# Teste: inicia o Agent escutando UDP na porta 8888
# (feche com Ctrl+C após confirmar que inicia)
cd ~/Micro-XRCE-DDS-Agent/
MicroXRCEAgent udp4 -p 8888

05 Mundos e modelos do Gazebo

Baixa e extrai os mundos e modelos criados para o projeto (plataforma UERJ, flares, torres, etc.) na pasta padrão de simulação. Eles convivem com os modelos originais do PX4 sem sobrescrevê-los.

bash
# Baixa o pacote de mundos/modelos do Google Drive (confirm=t p/ arquivo grande)
curl -L "https://drive.usercontent.google.com/download?id=1ZrOQYSJkF2iC9E3WfL-aPIoe1Xb52ykT&export=download&confirm=t" -o ~/simulation-gazebo.zip

# Extrai na pasta pessoal (cria/mescla ~/.simulation-gazebo)
unzip -o ~/simulation-gazebo.zip -d ~/

# Remove o zip para liberar espaço
rm ~/simulation-gazebo.zip
Dica Confira o tamanho com ls -lh ~/simulation-gazebo.zip antes de extrair — se o arquivo tiver poucos KB, o Drive devolveu a página de aviso em vez do arquivo (o parâmetro confirm é justamente para pular esse aviso em arquivos grandes, acima de ~100 MB).

06 Copiar os plugins compilados do Gazebo

Os plugins do Gazebo (câmera GStreamer, fluxo óptico, plataforma móvel) precisam ser os compilados nesta máquina — eles carregam um caminho interno (RUNPATH) que aponta para o build local do PX4.

bash
# Copia os .so gerados no build do PX4 para a pasta de plugins da simulação
cp ~/PX4-Autopilot/build/px4_sitl_default/src/modules/simulation/gz_plugins/lib*.so ~/.simulation-gazebo/plugins/
Por quê? Os plugins que vêm dentro do zip de mundos foram compilados em outra máquina, com o caminho /home/keyslav/… gravado no binário. Ao copiar os desta máquina, o caminho passa a existir e some o erro libOpticalFlow.so: cannot open shared object file.

07 Baixar e compilar os pacotes do projeto

Clona os quatro repositórios no workspace ROS 2, resolve dependências e compila.

bash
# Cria o workspace e clona os 4 pacotes (cada um na sua branch/versão)
mkdir -p ~/ros2_ws/src
cd ~/ros2_ws/src
git clone -b V1.0  https://github.com/Keyslav/drone_inspetor.git
git clone -b v1.0  https://github.com/Keyslav/drone_inspetor_msgs.git
git clone -b main  https://github.com/PX4/px4_msgs.git
git clone -b main  https://github.com/PX4/px4_ros_com.git
bash
# Resolve e instala as dependências ROS declaradas nos pacotes
cd ~/ros2_ws
sudo rosdep init
rosdep update
rosdep install --from-paths src --ignore-src -r -y

# Biblioteca de visão computacional usada pelos nós de detecção
pip install --break-system-packages ultralytics

# Compila o workspace (--symlink-install facilita editar scripts sem recompilar)
colcon build --symlink-install

# Carrega o workspace automaticamente em todo novo terminal
echo 'source ~/ros2_ws/install/setup.bash' >> ~/.bashrc
Nota Se o sudo rosdep init reclamar que já foi inicializado, ignore — basta seguir com rosdep update.
GPU Tem placa NVIDIA? Faça o passo 08 (GPU / CUDA) para rodar as redes de visão na GPU — bem mais rápido que na CPU.

08 GPU NVIDIA / CUDA (opcional)

As redes de visão computacional (YOLO / ultralytics) rodam muito mais rápido com uma GPU NVIDIA. Se a máquina não tiver GPU NVIDIA, pule este passo — tudo continua funcionando em CPU, apenas mais devagar.

Entenda Para rodar as redes você não precisa instalar o CUDA Toolkit completo: o PyTorch (instalado junto com o ultralytics no passo 07) já traz o runtime CUDA embutido. O único requisito real é o driver NVIDIA. O Toolkit (nvcc) só é necessário para compilar código CUDA próprio.
Antes de tudo: qual é o seu ambiente? O procedimento do driver é diferente em Ubuntu nativo e no WSL2 (Ubuntu dentro do Windows). Descubra com o comando abaixo e siga apenas a rota correspondente.
bash
# Se imprimir algo com "microsoft" ou "WSL", você está no WSL2
uname -r

Rota A — Ubuntu nativo (bare-metal)

Instale o driver NVIDIA dentro do próprio Ubuntu.

bash
# 1) Verifica se há GPU NVIDIA (vazio = sem GPU no barramento PCI)
lspci | grep -i nvidia

# 2) Mostra os drivers disponíveis e instala o recomendado
ubuntu-drivers devices
sudo ubuntu-drivers autoinstall

# 3) Reinicia para carregar o driver
sudo systemctl reboot

# 4) Após reiniciar, confirma o driver e a GPU
nvidia-smi
Dica A "CUDA Version" exibida no nvidia-smi é a versão máxima suportada pelo driver, não um toolkit instalado. Serve apenas como referência.

Rota B — WSL2 (Ubuntu no Windows)

Não instale driver no Ubuntu! No WSL2 o driver NVIDIA fica no Windows. Nunca rode ubuntu-drivers autoinstall nem apt install nvidia-driver-XXX dentro do Ubuntu — isso quebra o acesso à GPU do WSL. Basta manter o driver do Windows atualizado (GeForce/RTX no site da NVIDIA).
bash
# Único comando necessário: confirma que a GPU aparece via driver do Windows
nvidia-smi
O que é normal no WSL2 lspci | grep -i nvidia vem vazio e ubuntu-drivers não existe — é esperado, porque o WSL2 usa uma GPU paravirtualizada (/dev/dxg), sem passagem de PCI real. Se o nvidia-smi mostra a sua GPU (ex.: RTX A1000, CUDA Version 12.8), está tudo certo.

Confirmar que o PyTorch enxerga a GPU

Este teste usa o PyTorch instalado junto com o ultralytics no passo 07.

bash
# Deve imprimir "CUDA disponível: True" e o nome da GPU
python3 -c "import torch; print('CUDA disponível:', torch.cuda.is_available()); print(torch.cuda.get_device_name(0) if torch.cuda.is_available() else 'usando CPU')"
Se retornar False Em alguns casos o ultralytics instala uma versão do PyTorch só-CPU. Reinstale a versão com CUDA (comando abaixo) e rode o teste novamente.
bash
# Reinstala o PyTorch com suporte a CUDA 12.x
pip install --break-system-packages --force-reinstall torch torchvision --index-url https://download.pytorch.org/whl/cu124
Efeito colateral Essa reinstalação puxa o NumPy 2.x de volta, que quebra o cv2/cv_bridge e o scipy. Sempre que (re)instalar torch/ultralytics, rode o passo 11 depois para voltar ao NumPy 1.x: pip install --break-system-packages "numpy<2".

(Opcional) CUDA Toolkit completo

Só necessário se você for compilar código CUDA próprio (nvcc). Não é preciso para rodar o projeto.

bash
# OPCIONAL — instala o compilador CUDA (nvcc) e ferramentas de desenvolvimento
sudo apt install nvidia-cuda-toolkit -y

# Verifica a instalação do compilador
nvcc --version

09 Instalar o airframe 4030 no firmware

Registra o airframe de simulação do projeto (4030_gz_x500_uerj) dentro do PX4 e recompila.

bash
# Caminhos de origem (pacote) e destino (firmware)
SRC=~/ros2_ws/src/drone_inspetor/drone_inspetor/config/airframe
DST=~/PX4-Autopilot/ROMFS/px4fmu_common/init.d-posix/airframes

# Copia o arquivo do airframe para o firmware
cp "$SRC/4030_gz_x500_uerj" "$DST/"

# Registra o airframe no CMakeLists da v1.16 (insere a linha após 4001_gz_x500)
cd ~/PX4-Autopilot
sed -i '/4001_gz_x500$/a\	4030_gz_x500_uerj' ROMFS/px4fmu_common/init.d-posix/airframes/CMakeLists.txt

# Recompila o firmware já com o novo airframe
make px4_sitl
Correção Acrescentei o cd ~/PX4-Autopilot antes do sed — no rascunho o comando usava um caminho relativo sem estar na pasta certa. Não substitua o CMakeLists.txt inteiro por uma cópia: use o sed para inserir só a linha (versões diferentes do arquivo quebram o build com erros como 1060_gazebo-classic_rover doesn't exist).
Verificação Confirme a inserção com grep 4030 ROMFS/px4fmu_common/init.d-posix/airframes/CMakeLists.txt — a linha deve estar indentada com tab, igual às vizinhas.

11 Corrigir o conflito do NumPy

O ultralytics e o PyTorch instalam o NumPy 2.x, que é incompatível com o cv2/cv_bridge (compilados contra o NumPy 1.x) e com o scipy. A correção é fixar o NumPy em 1.x — o torch, o torchvision e o ultralytics funcionam normalmente com ele.

bash
# Rebaixa o NumPy para a série 1.x (garante uma versão compatível instalada)
pip install --break-system-packages "numpy<2"
Execute por último Rode este passo depois de qualquer pip install de ultralytics, torch ou torchvision (incluindo o fallback do passo 08) — esses pacotes reintroduzem o NumPy 2.x toda vez.
Verificação Confirme que o NumPy ativo é 1.x e que tudo importa junto: python3 -c "import numpy, torch; print(numpy.__version__, torch.cuda.is_available())". A versão deve começar com 1. (ex.: 1.26.4).
Alternativa Se preferir apenas remover o NumPy do pip e usar o do sistema (python3-numpy, 1.26.x no Ubuntu 24.04): python3 -m pip uninstall -y numpy --break-system-packages.

12 QGroundControl (estação de solo)

Interface de controle e telemetria. Detecta o PX4 automaticamente por UDP (porta 14550) quando a simulação está rodando.

bash
# Dá acesso às portas seriais (necessário p/ drone real via USB)
sudo usermod -aG dialout "$(id -un)"

# Desativa o ModemManager, que pode "sequestrar" a porta serial do QGC
sudo systemctl mask --now ModemManager.service

# Dependências de vídeo (GStreamer) e de interface gráfica
sudo apt install gstreamer1.0-plugins-bad gstreamer1.0-libav gstreamer1.0-gl -y
sudo apt install python3-gi python3-gst-1.0 -y
sudo apt install libxcb-xinerama0 libxkbcommon-x11-0 libxcb-cursor-dev -y

# Suporte a AppImage (no Ubuntu 24.04 o pacote é libfuse2t64)
sudo apt install libfuse2t64 -y
Correção No Ubuntu 24.04 o pacote FUSE chama-se libfuse2t64 (não libfuse2). Sem ele, o AppImage falha com error loading libfuse.so.2.
bash
# Baixa o AppImage do QGroundControl (-L segue redirecionamentos, -o salva o arquivo)
cd && mkdir -p QGC && cd QGC
curl -L "https://d176tv9ibo4jno.cloudfront.net/builds/master/QGroundControl-x86_64.AppImage" -o QGroundControl-x86_64.AppImage

# Torna o AppImage executável
chmod +x QGroundControl-x86_64.AppImage

# Executa o QGroundControl
./QGroundControl-x86_64.AppImage
Correção O curl original não tinha -L nem -o, então ele apenas jogava a resposta na tela sem salvar o arquivo. Adicionei -L (segue redirecionamentos) e -o (grava no disco).
Nota A adição ao grupo dialout só vale após logout/login. Para SITL isso não é crítico (a conexão é por UDP), mas é necessário para drone real via USB.

13 Execução do projeto

Com tudo instalado, o projeto sobe em cinco terminais. Abra um terminal novo para cada um (todos já carregam ROS 2 e o workspace pelo .bashrc).

Terminal 1 — Mundo físico (Gazebo, sem visual)

Inicia a simulação do mundo no Gazebo em modo headless (sem janela gráfica).

bash
bash ~/ros2_ws/src/drone_inspetor/drone_inspetor/scripts/projeto_gazebo.sh
Terminal 2 — Visualização gráfica (opcional)

Abre a janela do Gazebo (cliente GUI) conectada à simulação do Terminal 1.

bash
gz sim -g
Dica Em máquinas com GPU NVIDIA, use o script do pacote que força a renderização na NVIDIA: bash ~/ros2_ws/src/drone_inspetor/drone_inspetor/scripts/gz_sim.sh
Terminal 3 — Firmware PX4 + drone

Inicia o PX4 e insere o drone no mundo Plataforma_UERJ usando o airframe 4030.

bash
cd ~/PX4-Autopilot
SIM_GZ_HOME_LAT=-22.633890 SIM_GZ_HOME_LON=-40.093330 SIM_GZ_HOME_ALT=0 GZ_IP=127.0.0.1 PX4_GZ_STANDALONE=1 PX4_GZ_WORLD=Plataforma_UERJ PX4_SYS_AUTOSTART=4030 PX4_GZ_MODEL_POSE="-70,-27,57" ./build/px4_sitl_default/bin/px4
Terminal 4 — Middleware uXRCE-DDS

Sobe a ponte que conecta o PX4 ao ROS 2 (porta UDP 8888).

bash
MicroXRCEAgent udp4 -p 8888
Complemento Este terminal estava em branco no rascunho — o comando correto é o MicroXRCEAgent udp4 -p 8888 (mesmo do teste do passo 04).
Terminal 5 — Aplicação do projeto

Lança a estação de solo do projeto (detecção, missão e controle).

bash
ros2 launch drone_inspetor ground_station_launch.py
Ordem de subida Suba na ordem 1 → 3 → 4 → 5 (o Terminal 2 é opcional e pode entrar a qualquer momento). O PX4 (T3) só encontra o mundo se o Gazebo (T1) já estiver rodando.
